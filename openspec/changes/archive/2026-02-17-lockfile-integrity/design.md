## Context

The current Dockerfile generated by `GenerateDockerfile()` uses `--no-lock` everywhere. The engine ships with a `deno.lock` file (generated by `deno cache` during development), but it is not copied into the container image. This means builds are not reproducible and dependency integrity is not verified.

## Goals / Non-Goals

**Goals:**
- Pin engine dependencies at build time by copying and using `deno.lock`
- Verify dependency integrity during the `deno cache` build step

**Non-Goals:**
- Locking workflow node dependencies (they are mounted via ConfigMap and have no separate lock file)
- Runtime lock checking (the filesystem is read-only and the cache is already verified at build)

## Decisions

1. **COPY deno.lock before deno cache** -- The lock file must be present before the cache step runs so Deno can verify checksums. Place the COPY after the existing `COPY .engine/deno.json` line.

2. **`--lock=deno.lock` on cache, `--no-lock` on runtime** -- Build step verifies integrity. Runtime keeps `--no-lock` because: (a) the root filesystem is read-only, so Deno cannot update the lock file; (b) the cache was already verified at build time.

## Risks / Trade-offs

- [Risk] If `deno.lock` is missing from the build context, the Docker build will fail. --> Mitigation: The engine scaffold already generates `deno.lock`. The `tntc build` process includes the `.engine/` directory which contains the lock file.
